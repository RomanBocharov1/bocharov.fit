ТЕСТ-ДИЗАЙН - этап процесса тестирование ПО, на котором создаются тест-кейсы на основании техник, которые уменьшают количество и время тестов



эквивалентные классы - нужна, чтобы не тестить большие обьемы входящих данных
разбиваем тестовые данные на эквивалентные классы
эквивалентные классы - набор подзначений, с которыми предположительно система должна вести себя одинаково
Пример:
фитнес трекер от 1000 шагов до 10000 шагов, при наборе определенного кол-ва шагов трекер выдает сообщение
1.до 1000
2.от 1001 до 5000
3.от 5001 до 10000
1,2,3 это эквивалентные классы ( у каждого класса есть граница)
значение (количество шагов) разбиваем на эквивалентные классы



метод граничных значений(граничные значения фигурируют в коде if x>5) - у эквивалентных классов есть начало и конец , граничные значения (-1,0,1 и 99,100,101)
1.сначала узнаем какие значения принимает система - по 1, 0.5, 5 и какие требования были
2.потом узнаем , где у эквивалентного класса граница
3.потом узнаем , величину шага между соседними значениями (1, 0.5, 5)
техника нужна там, где есть классы еквивалентности и на границах они соприкасаются



анализ перехода состояний системы(state transition) - нужна , чтобы сформировать тест-кейсы так, чтобы не упустить переходы и триггеры
проверить все переходы системы из одного состояние в другое и что невозможные переходы не состоятся
-ввести валидный пин в банкомат
-ввести невалидный пин
-ввести невалидный 3 раза и блок
-ввести невалидный 2 раза и нажать отмена
(крч,по нашей системе, рисуем страницы и переходы от нее возможные и невозможные)

Тестовые артефакты - инстументы, используемые в тестировании.

Тест план - документ , который описывает обьем работ( описание обьекта,стратегии ,
расписания,критериев начала и конца тестирования,используемого оборудования,
инстументальных стредств,срец.знаний,оценка рисков с вариантами их решения) Дополнить!!!!!!

Чек-лист - документ, который описывает какие функции должны быть проверены.

Тест-кейс - артефакт, описывающий шаги, условия, параметры, необходимые
для проверки функции или ее части(на основании ТЗ)
Бывают позитивные и негативные.
(Вопрос на собесе: Сколько нужно тест-кейсов для проверки функционала?
Ответ: 1)
Содержимое тест-кейса:
1.ID
2.Приоритет
3.Описание(заголовок)
4.Предусловие - список шагов, которые нужно сделать до начала теста
(например, авторизоваться,прежде чем создавать группу в вк)
5.Модуль
6.Шаги
7.Ожидаемый результат - Фактический результат
8.Статус тест-кейса- есть баг или нет
9.Автор
Требования к тест-кейсам:
- отсутствие зависимости друг от друга
- четкие формулировки
-наличие детальной, но не избыточной информации
-дефект должен быть очевиден
Тест -кейс проверяет одну конкретную вещь
в шагах указывать вводим валидные данные или невалидные данные по шагам ( вводим буквы -
валидное значение, вводим спец.символы - невалидное значение)
в тест кейсе должна быть вся инфа для его проведения
ожидаемый результат должен быть у каждого шага проверки
пруфов не должно быть, иначе при изменении интерфейса нужно будет делать снова???
заголовок - что проверяем, описание функционала: "Отправка сообщения через форму обратной связи на странице контакты"

S2 | Major - баг,связанный с не основными функциями ПО

На примере машины:
S1 - поцарапана краска

S3 - едет через раз
